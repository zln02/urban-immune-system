Urban Immune System - 중간발표 통합 설명 자료
(모르는 단어 없이 처음 보는 사람도 이해할 수 있는 버전)


1. 이 프로젝트가 뭔지 한 줄로
------------------------------------------------

"감기·독감이 크게 퍼지기 1~3주 전에 미리 알려주는 도시 면역 시스템"

왜 만들었나요?

지금 문제: 환자가 증상 → 병원 방문(2~4일) → 진단·검사(1~3일) → 의료기관 보고(1~5일) → 질병관리청 집계(3~7일) = 최소 2주 후에야 유행 사실을 알 수 있어요.

비유: 집에 불이 난 지 2주 후에야 소방서에 신고가 접수되는 것

우리 해결책: 병원 가기 전에 나타나는 신호 3가지를 동시에 감시해서 1~3주 먼저 경보를 줍니다.

비유: 기상청은 비가 오고 나서 "비 왔습니다"라고 하지 않아요. 구름·기압·온도를 미리 보고 예보합니다. 우리도 똑같아요.


2. 3가지 신호 (Layer) 설명
------------------------------------------------

[Layer 1] 약국 OTC - 1~2주 먼저 탐지
(Over-the-Counter, 처방전 없이 사는 약)

사람들은 병원 가기 전에 먼저 약국에서 감기약·해열제를 삽니다. 이 구매량을 네이버 쇼핑인사이트 API로 수집합니다.

숫자 근거: 2025~26 독감 시즌 실데이터에서 약 구매 급증 시점이 질병관리청 임상 신고보다 평균 2주 빨랐어요.

비유: 편의점에서 우산이 갑자기 많이 팔리면 비가 올 것


[Layer 2] 하수 바이러스 - 2~3주 먼저 탐지
(Wastewater-based epidemiology, KOWAS)

감염된 사람의 소변·대변에 바이러스가 섞여 하수도로 흘러갑니다. 증상이 없어도 바이러스는 이미 배출되기 때문에 병원 기록보다 가장 빨리 탐지돼요. 가장 신뢰도 높은 신호.

데이터 출처: 질병관리청 KOWAS PDF (17개 시도 x 56주)

비유: 강 상류의 오염을 측정해서 하류 마을 건강을 예측


[Layer 3] 검색 트렌드 - 1~2주 먼저 탐지
(Naver DataLab)

"독감 증상", "열이나요", "기침" 같은 검색어가 갑자기 많아지면 유행 징조예요. 아프면 병원 전에 먼저 검색하는 행동 패턴을 이용합니다.

주의: 단독 사용 금지! 2013년 구글이 검색어만 믿었다가 실제보다 2배 과대 예측하고 서비스를 종료했어요. 뉴스에 독감 기사 많아지면 검색도 많아져서 AI가 착각하기 때문이에요.

비유: SNS에서 "기침 심하다"는 글이 갑자기 늘어나는 것


핵심 안전장치: 3개 중 2개 이상이 동시에 올라야 경보!
- 1개만 올라가면 무시 (오경보 방지)
- 왜 2개? 1개 기준은 오경보 많음, 3개 모두는 탐지를 너무 많이 놓침. 실험 결과 2개 기준이 최적.

비유: 교통사고 목격자 3명 중 2명 이상이 같은 말을 해야 신고하는 것

종합 점수 = (약국 x 35%) + (하수도 x 40%) + (검색 x 25%)
- 하수도 비중이 제일 높은 이유: 증상 없어도 잡히는 객관적 증거


3. 경보 단계
------------------------------------------------

GREEN  : 종합점수 30 미만 / 안전, 걱정 없음
YELLOW : 30~55 / 주의, 손 씻기·마스크 챙겨두기
ORANGE : 55~75 / 경고, 노약자 외출 자제
RED    : 75 이상 / 위험, 긴급 조치 필요

대시보드 예시 (2025~26 백테스트 시나리오)
- 서울 48.3점 → YELLOW
- 대전 60.8점 → ORANGE
- 부산 78.3점 → RED

비유: 공기질 지수(AQI)처럼 복잡한 데이터를 숫자 하나로 요약한 것


4. AI 성능 지표 용어
------------------------------------------------

F1-Score = 0.667
Precision(정밀도)과 Recall(재현율)을 합친 종합 점수. 0~1 사이, 1에 가까울수록 완벽.
계산식: 2 x (1.000 x 0.500) / (1.000 + 0.500) = 0.667
비유: 시험에서 맞힌 개수와 틀린 개수를 동시에 고려한 종합 점수
→ 100점 만점에 67점 (목표 70점에 근접)


Precision (정밀도) = 1.000 → 오경보 0건!
"경보를 울렸을 때 실제로 유행이 온 비율"
계산: TP / (TP + FP) = 8 / (8 + 0) = 1.000
우리 시스템이 경보 울린 것 100%가 실제 유행이었어요.
비유: 화재 경보기가 10번 울렸는데 10번 모두 진짜 불


Recall (재현율) = 0.500
"실제 유행이 왔을 때 경보를 울린 비율"
계산: TP / (TP + FN) = 8 / (8 + 8) = 0.500
실제 유행 기간의 절반을 잡았어요. 오경보 0건을 위해 엄격한 기준을 쓴 결과.
비유: 범인 10명 중 5명을 잡았는데, 잡은 5명은 전부 진짜 범인


AUC-ROC = 0.931
"유행이냐 아니냐를 구별하는 능력" (0~1, 0.5는 동전 던지기 수준)
혈액검사로 병을 진단할 때 정상/이상을 얼마나 잘 구별하는지와 같아요.
0.931 → 매우 우수한 수준


혼동행렬 (TP / FP / FN / TN)
AI 예측 결과를 4칸으로 나눈 성적표예요.

           | 실제 유행 있음     | 실제 유행 없음
-----------|--------------------|----------------
경보 울림  | TP = 8 (잘 잡음)   | FP = 0 (헛경보)
경보 안함  | FN = 8 (놓침)      | TN = 5 (맞게 조용)

FP = 0 → 오경보 0건이 가장 큰 자랑
비유: 받아쓰기 채점표 - 맞게 썼는지, 틀리게 썼는지 4칸으로 나눈 것


FAR (False Alarm Rate) = 0.557
"유행이 아닌 기간에 경보를 낸 비율" (낮을수록 좋음, 개선 목표)


Lead Time (선행 시간) = 평균 5.9주
임상 확진자가 폭발하기 약 6주 전에 경보를 낸다는 뜻.
17개 시도 백테스트 평균 기준. 가장 빠른 지역: 세종(9주 선행)
비유: 태풍이 상륙하기 6주 전에 예보가 나오는 것


17개 시도 일반화 테스트
처음 4개 지역으로 만든 모델이 나머지 13개 지역에서도 작동하는지 확인.
4지역 Recall 0.906 → 17지역 0.838 → 크게 떨어지지 않음 → 전국 확장 가능
비유: 서울 레시피로 만든 음식이 부산에서도 똑같이 맛있는 것


5. 통계 검증 용어
------------------------------------------------

Walk-forward 교차검증 (시간 순 교차검증)
AI를 테스트할 때 절대 미래 데이터를 못 보게 하는 방법.
예: 1~10주 데이터로 학습 → 14~16주 데이터로 평가 (4주 공백)
왜 4주 공백? 우리 시스템 최대 선행이 3주라서, 4주를 비워야 미래를 "슬쩍 보고 맞히는" 커닝이 차단됩니다.
비유: 내일 시험 문제지를 미리 보지 않고, 공부한 실력만으로 보는 것


Granger 인과검정 (p < 0.05)
"A 신호가 B보다 먼저 움직이는가?"를 수학으로 증명하는 방법.
p < 0.05 = 이 결과가 우연일 확률이 5% 미만 → 통계적으로 유의미.
우리 결과: 3개 Layer 모두 p < 0.05 통과 / 검색 신호 임상 신고보다 3주 선행 (p=0.007)
비유: 닭이 달걀보다 먼저냐를 수학으로 증명하는 것


CCF 히트맵 (Cross-Correlation, 교차 상관)
두 신호가 시간적으로 얼마나 앞뒤로 연관되는지 색깔로 보여주는 그래프.
빨간색 = 강한 상관, Lag(주) 축 = 몇 주 앞서는가.
슬라이드 해석: L2 하수 Lag +2~3주에서 빨간색 → 하수가 2~3주 앞서 올라감
비유: 태풍 이동 경로를 그려서 어느 지역에 언제 영향을 줄지 보여주는 지도


백테스트 (Backtest)
모델을 만들 때 쓰지 않은 과거 실데이터로 성능을 검증하는 방법.
사용한 데이터: KCDC 임상 신고 1,020건 (17개 시도 x 60주 정답지)
비유: 기상 모델을 "작년 날씨 데이터"로 돌려서 "그때 맞았냐?" 확인하는 것


재현 가능성 (Reproducibility)
누가 코드를 돌려도 발표에서 보여드린 숫자가 그대로 나온다는 것.
4개 명령어로 재현: KOWAS 952건 수집 → 네이버 1,938건 수집 → ML 검증 실행 → DB 확인
비유: 레시피대로 만들면 누가 요리해도 같은 맛이 나는 것


6. AI 모델 용어
------------------------------------------------

TFT (Temporal Fusion Transformer)
시간 흐름 데이터로 미래를 예측하는 딥러닝 AI 모델.
7일·14일·21일 후 감염병 위험도를 계산해요.
"왜 그런 예측을 했는지 이유를 설명"할 수 있어서 정부 납품에 유리.
현재 완성도: 55% (코드·서빙 완성, 실데이터 연결 중)
비유: 날씨 예보처럼 과거 기온 패턴으로 다음 주 날씨를 계산하는 AI


Autoencoder (오토인코더) - 이상 탐지 AI
평상시 신호 패턴을 학습한 뒤, 갑자기 다른 패턴이 나타나면 경보.
TFT가 예측이라면, 오토인코더는 이상 감지 담당.
비유: 집 평소 소음을 다 외웠다가 낯선 소리가 나면 알람을 울리는 스마트홈 센서


RAG (Retrieval-Augmented Generation, 래그)
AI가 답변 전에 공식 문서를 먼저 찾아보고 그 내용을 근거로 답하는 방법.
출처: WHO·ECDC·질병관리청 가이드 문서 5건.
"이 내용은 WHO 2024 가이드 3페이지에서 가져왔습니다"처럼 출처 표시.
정부 납품에서 "AI가 어디서 가져온 말이냐?"를 답할 수 있어야 납품 가능.
비유: 오픈북 시험. 외운 것만 말하는 일반 AI와 달리 문서를 펴놓고 답함.


LLM (Large Language Model, 대형 언어 모델)
ChatGPT, Claude 같은 대화형 AI. 엄청난 양의 글을 학습해서 사람처럼 문장을 쓰고 질문에 답합니다.
우리 시스템은 Claude를 써서 경보 리포트를 한국어로 자동 작성해요.


환각 (Hallucination)
AI가 사실이 아닌 것을 사실처럼 지어내는 현상.
RAG를 쓰면 공식 문서를 근거로만 답해서 환각을 줄일 수 있어요.


임베딩 / 384차원 벡터 / 코사인 유사도
임베딩 = 문장을 숫자 384개짜리 좌표로 바꾸는 것. 의미가 비슷한 문장은 좌표도 비슷해요.
코사인 유사도 = 두 문서가 얼마나 비슷한지 0~1 점수로 계산.
AI가 질문을 받으면 가장 유사한 문서 3개를 골라 참고.
비유: "기침", "콧물", "열" 세 단어가 비슷한 좌표에 모이는 3D 지도


7. 데이터·인프라 기술 용어
------------------------------------------------

API (에이피아이)
프론트엔드와 백엔드가 대화하는 약속된 규칙.
"이 주소로 요청하면 이 데이터를 줄게"라는 메뉴판.
비유: 카페 메뉴판 - "아이스 아메리카노 주세요(요청)" → "여기요(응답)"


OCR / 픽셀 분석
OCR = 이미지 속 글자를 컴퓨터가 읽는 기술 (가끔 오타 발생).
우리는 OCR 대신 차트 막대의 색깔(RGB)을 픽셀 단위로 분석.
코로나(파랑), 인플루엔자(주황), 노로(노랑) → 색으로 구분해 높이 계산.
성과: 17개 시도 x 56주 = 952건 자동 추출, 수동 입력 0건.
비유: 그래프 사진에서 눈대중 대신 컴퓨터가 픽셀 색깔을 정밀 측정하는 것


TimescaleDB (타임스케일디비)
시간 순서에 특화된 데이터베이스.
"서울시 2025년 11월 3주차 감기약 구매량"처럼 날짜+데이터를 빠르게 조회.
비유: 일반 서랍장(DB) vs 날짜별로 칸이 나뉜 달력형 가계부(TimescaleDB)


Kafka (카프카)
여러 곳에서 오는 데이터를 한 곳으로 모아 순서대로 전달하는 메시지 허브.
(현재 프로젝트는 발표 안정성을 위해 DB 직접 저장 방식 사용 중)
비유: 약국·하수·검색 데이터를 한 번에 모아주는 택배 물류 허브


SSE (Server-Sent Events)
서버가 브라우저에 데이터를 한 조각씩 실시간으로 흘려보내는 방식.
Claude가 리포트를 쓰는 동안 글자가 한 글자씩 화면에 나타나는 효과.
첫 응답 500ms 이내.
비유: ChatGPT에서 답이 타이핑되듯 나오는 그 효과


Docker / Kubernetes / GKE
Docker = 프로그램과 실행 환경을 하나의 박스(컨테이너)에 포장. "내 컴퓨터에서는 되는데 다른 컴퓨터에서 안 돼요" 문제 해결.
Kubernetes = 그 박스 수십 개를 자동으로 관리 (죽으면 재시작, 과부하면 서버 추가).
GKE = 구글 클라우드에서 운영하는 Kubernetes 서비스.
비유: 여러 택배 박스를 자동 분류·배송하는 스마트 물류 센터


FastAPI / async (비동기)
FastAPI = 파이썬으로 만드는 웹 API 서버 프레임워크.
async = 여러 요청을 동시에 처리하는 방식.
비유: 여러 테이블 주문을 동시에 처리하는 능숙한 웨이터


GitHub Actions / CI·CD
코드를 깃허브에 올리면 자동으로 테스트 실행 → 통과하면 자동 서버 배포.
CI = 테스트 자동화 / CD = 배포 자동화. 현재 pytest 22개 테스트 자동 실행.
비유: 원고 제출하면 자동 교정→인쇄→배송까지 해주는 시스템


Streamlit / Next.js
Streamlit = 파이썬 코드만으로 빠르게 만드는 데이터 대시보드 (프로토타입).
Next.js = 정식 웹 대시보드. 지도·챗봇·실시간 그래프 등 고급 기능 구현.
비유: Streamlit = 기능만 확인하는 빠른 시제품 / Next.js = 정식 출시 앱


PR (Pull Request)
내가 만든 코드를 팀 코드에 합치기 전 "검토해줘!" 요청.
PR #4처럼 번호가 붙어서 기능 추가 기록이 남아요.
비유: 레포트 제출 전에 선생님께 먼저 확인받는 것


8. 팀원별 역할 설명
------------------------------------------------

박진영 - PM(팀장) + AI 엔진

PM이 하는 일: 전체 방향 설정, 일정 조율, 코드 리뷰, 팀원 PR 머지 관리. 개발도 직접 함.
AI 담당: TFT 예측 모델, Autoencoder 이상탐지, 3-Layer 앙상블(교차검증) 알고리즘 설계.
실제 한 것: 약국·검색 1년치 자동 수집, 하수 PDF 파이프라인, XGBoost 예측 모델 학습, RAG 리포트 엔진 구현, 전국 17개 시도 확장.
비유: 영화 감독. 전체 스토리 설계하고 팀원들이 각자 맡은 씬을 잘 찍도록 지휘.

---

이경준 - 백엔드 API + DB

백엔드가 하는 일: 화면에 안 보이는 서버 쪽 코드. "데이터 줘!"에 DB에서 꺼내 전달.
실제 한 것 (PR #4):
- /alerts/current → 지역별 현재 경보 레벨 제공
- /alerts/regions → 전국 17개 시도 동시 조회
- /alerts/stream  → AI 리포트 실시간 스트리밍(SSE)
- SQL 파라미터 바인딩 → 해킹 방어 (ISMS-P 준수)
비유: 식당 주방장. 손님(화면)이 주문하면 냉장고(DB)에서 재료 꺼내 요리해서 전달.

---

이우형 - 데이터 엔지니어 (파이프라인)

하는 일: 외부에서 데이터를 자동 수집해 DB에 저장. AI가 분석할 재료를 끊임없이 공급.
실제 한 것:
- KOWAS PDF 자동 다운로드 → 픽셀 분석 → DB 저장
- 17시도 x 56주 = 952건 자동 추출 (수동 0건)
- 네이버 쇼핑·데이터랩 API 수집 파이프라인
- TimescaleDB 스키마 설계 / Docker Compose 환경 구성
비유: 마트 물류 담당자. AI(요리사)가 요리할 수 있도록 전국 농장(외부 API)에서 재료를 매일 가져다 놓음.

---

김나영 - 프론트엔드 (화면 개발)

하는 일: 사용자가 직접 보는 화면(웹 페이지). 버튼, 지도, 그래프, 레이아웃 등 눈에 보이는 전부.
실제 한 것:
- 전국 17개 시도 지도에 색깔로 위험도 표시
- Claude AI 경보 리포트 실시간 타이핑 화면
- 챗봇 위젯 (대시보드 안에서 AI에게 질문 가능)
- KPI 카드, Layer별 신호 카드, 이상탐지 패널
비유: 인테리어 디자이너. 건물 구조는 다른 팀이 짓고, 나영 씨는 사람이 보고 쓰는 공간을 꾸밈.

---

박정빈 - DevOps + QA + 법 + 디자인

DevOps: Docker 컨테이너 설정, GitHub Actions 자동 배포.
QA: pytest 22개 테스트로 버그 자동 감지.
법: 정부 납품용 ISMS-P 인증 준비, 개인정보처리방침.
디자인: 색맹 안전 팔레트(Okabe-Ito), 발표 슬라이드 제작.
비유: 건물 관리자 + 안전검사원 + 법무팀. 서비스가 무너지지 않게 유지하고 법 기준을 통과시킴.


9. 사업화 용어
------------------------------------------------

B2G (Business to Government)
정부·공공기관에 납품하는 사업 모델.
목표: 질병관리청, 광역지자체 보건소에 시스템 납품.
가격: 지자체 연 2,000~6,000만원 / 광역 연 1~3억원.
비유: 학교 급식 납품업체가 작은 학교 1곳 시험 납품 후 전국 확장하는 것

6개월 내 목표: 지자체 PoC 1건, 정부과제 수주 2,000~5,000만원.


ISMS-P 인증
정부기관 납품 시 필수 정보보호 인증. "개인정보를 안전하게 관리하고 있음"을 국가가 공식 인증.
비유: 식당이 위생 등급을 받아야 학교 급식 납품이 가능한 것처럼


SCI 논문

Phase 3 (2027 상반기) 목표: SCI 논문 1편.


10. 예상 Q&A
------------------------------------------------

Q. 왜 하수도 비중이 40%로 제일 높나요?
A. 증상이 없어도 바이러스가 하수로 나와요. 무증상 감염자까지 잡을 수 있는 유일한 객관적 신호라서 가장 신뢰도가 높습니다.

Q. 검색어만으로 경보 내면 안 되나요?
A. 안 됩니다. 구글이 2013년 검색어만 믿다가 2배 과대예측으로 폐지됐어요. 뉴스에 독감 기사 많아지면 검색도 많아져서 AI가 착각하기 때문이에요.

Q. Recall이 0.5로 낮은데 괜찮나요?
A. 오경보 0건(Precision=1.0)을 유지하기 위한 선택입니다. 정부 납품에서 "헛경보"가 반복되면 신뢰를 잃어요. 경보 기준을 낮추면 Recall이 0.85 이상으로 올라가지만 오경보도 생깁니다.

Q. 실제 데이터인가요?
A. 약국·검색은 네이버 실 API, 하수도는 질병관리청 실 PDF 데이터예요. 검증은 2025~26 독감 시즌 실데이터 기반 백테스트로 수행했습니다.
