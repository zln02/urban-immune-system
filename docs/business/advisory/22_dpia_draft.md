# 개인정보 영향평가 (DPIA) 초안 — Urban Immune System

> 본 문서는 KISA ISMS-P 사전 컨설팅 자문 자료로 첨부됩니다.
> 정식 DPIA는 Phase 3 (2027 H1) 인증 신청 시 갱신 예정.

## 1. 시스템 개요

| 항목 | 내용 |
|---|---|
| 시스템명 | Urban Immune System (UIS) v0.4 |
| 목적 | 인플루엔자 등 감염병 조기경보 (보건당국 의사결정 보조) |
| 운영 주체 | 동신대학교 컴퓨터공학과 캡스톤 팀 (학생 PoC) |
| 향후 운영 | 질병관리청 · 광역 자치단체 (Phase 3 이후) |
| 데이터 보유 기간 | 원시 신호 7일 / 집계 점수 무기한 (시·도 단위) |

## 2. 처리하는 데이터 분류

### 2.1 처리 대상 (개인정보 비해당)

| 데이터 | 출처 | 식별 가능성 | 분류 |
|---|---|---|---|
| 약국 OTC 트렌드 지수 | 네이버 쇼핑인사이트 (시·도 단위 집계 API) | **불가** | 통계 데이터 |
| 하수 바이러스 농도 | KOWAS 공개 PDF (지역 평균값) | **불가** | 환경 데이터 |
| 검색 트렌드 지수 | 네이버 데이터랩 (시·도 단위 집계 API) | **불가** | 통계 데이터 |
| 기상 (기온·습도) | 기상청 KMA (관측소 단위) | **불가** | 환경 데이터 |

### 2.2 처리 대상 아님 (절대 수집하지 않음)

- 개인 식별자 (이름·주민번호·전화번호·이메일·IP)
- 의료 기록 (진단명·처방·검사 결과)
- 위치 정보 (GPS·동선)
- 결제 정보 (카드번호·계좌)

## 3. 개인정보 보호법 제2조 제1호 적용 여부

> 개인정보보호법 제2조 1호: "살아 있는 개인에 관한 정보로서 성명·주민등록번호 및 영상 등을 통하여 개인을 알아볼 수 있는 정보. 다만, 해당 정보만으로는 특정 개인을 알아볼 수 없더라도 다른 정보와 쉽게 결합하여 알아볼 수 있는 것을 포함."

**판단 (자체 검토 — KISA 자문 요청 대상)**:

본 시스템이 처리하는 모든 입력 데이터는 출처 단계에서 시·도 단위 집계 지수로 변환된 후 수신됩니다. 따라서:

- 개인 식별 직접 처리 없음 ✅
- 다른 정보와 결합해도 시·도 인구 (수십만~수천만) 중 특정 개인 식별 불가 ✅
- 본 시스템 단독으로는 개인정보보호법 제2조 1호 비해당으로 판단

→ **KISA 자문 의뢰 사항** (2번 — 11_kisa_consult_application.md 참조).

## 4. 적용된 보안 통제

### 4.1 기술적 통제

| 통제 | 구현 위치 |
|---|---|
| 모든 입력 환경변수 검증 (Pydantic) | `backend/app/config.py` |
| API 키 .env 격리 + 코드 하드코딩 차단 (pre-commit) | `.pre-commit-config.yaml` |
| K8s Pod 비루트 실행 (`runAsNonRoot: true`) | `infra/k8s/*-deployment.yaml` |
| 권한 상승 차단 (`allowPrivilegeEscalation: false`) | 동일 |
| 읽기 전용 루트 파일시스템 | 동일 |
| Linux capability 모두 drop | 동일 |
| Kafka 메시지 손실 방지 (`acks=all` + `retries=3`) | `pipeline/collectors/kafka_producer.py` |
| SSE 단방향 (쿠키 인증 미사용) | `backend/app/api/alerts.py` |
| AI 면책 강제 삽입 (ISMS-P 2.9 / EU AI Act 13·14) | `ml/rag/report_generator.py` |
| 감사 로그 (`triggered_by`, `trigger_source`) | `infra/db/init.sql` |

### 4.2 관리적 통제

| 통제 | 상태 |
|---|---|
| .env 파일 git 커밋 차단 | ✅ `.gitignore` |
| API 키 로그 출력 금지 정책 | ✅ 코드 리뷰 가이드 |
| pytest 보안 테스트 (config·k8s 보안 컨텍스트) | ✅ `tests/test_k8s_security.py` |
| 정기 의존성 취약점 스캔 | ⏳ Phase 2 (Dependabot 활성화 예정) |

### 4.3 미구현 (Phase 2~3 예정)

- DB 컬럼 단위 암호화 (AES-256)
- KMS 키 관리 (현재 .env 평문 — 운영 단계 필수)
- 정기 침투 테스트
- ISMS-P 정식 심사

## 5. 위험도 평가

| 위험 시나리오 | 가능성 | 영향도 | 통제 |
|---|---|---|---|
| 네이버 API 키 유출 → 외부 호출 | 낮음 | 중 | .env 격리 + pre-commit + git 커밋 차단 |
| TimescaleDB 직접 액세스 → 트렌드 지수 노출 | 낮음 | **낮음** (개인정보 비해당) | K8s 네트워크 정책 + DB 인증 |
| RAG 리포트의 잘못된 의료 권고 | 중 | 높음 | 면책 조항 강제 + 인간 전문가 검토 전제 |
| 시스템 다운 → 경보 미발령 | 중 | 중 | Kafka 7일 재처리 + 실패 시 이전 값 유지 |

## 7. KISA 자문 대상 결정 사항

1. 시·도 단위 집계만 처리하는 시스템의 개인정보보호법 제2조 1호 비해당 판정
2. ISMS-P 신청 시 본 시스템 규모(연 동시접속 ~50명) 권고 통제 범위
3. AI 면책 조항이 의료기기법 SaMD 비해당 근거로 충분한지
4. 네이버 API 응답을 정규화 지수로만 변환·저장하는 방식의 약관 적합성
