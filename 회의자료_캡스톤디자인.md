## 1. 프로젝트 현황 요약

### 1.1 핵심 아이디어
의료 데이터(확진 집계)에 의존하지 않고, **3개의 비의료 신호**를 교차검증해 인플루엔자를 **1~3주 선행 감지**하는 시스템.

| Layer | 데이터 소스 | 선행시간 | 출처 |
|-------|-----------|---------|------|
| L1: 약국 OTC | 감기약/해열제 구매 트렌드 | 1~2주 | 네이버 쇼핑인사이트 API |
| L2: 하수 바이오마커 | 인플루엔자 바이러스 농도 | 2~3주 | KOWAS (환경부) PDF |
| L3: 검색어 트렌드 | 독감 증상/타미플루 검색량 | 1~2주 | 네이버 데이터랩 API |

### 1.2 검증된 성과 (2024-25 시즌, 26주 실데이터)

| 모델 | Precision | Recall | F1-Score | 오경보 |
|------|-----------|--------|----------|--------|
| 약국 단독 | 0.00 | 0.00 | 0.00 | 0건 |
| 하수 단독 | 1.00 | 0.44 | 0.62 | 0건 |
| 검색 단독 | 0.67 | 0.22 | 0.33 | 2건 |
| **3-Layer 통합** | **1.00** | **0.56** | **0.71** | **0건** |

- Granger 인과검정: 3개 Layer 모두 p < 0.05 유의
- 핵심 가치: **오경보 0건** (Precision 1.00) — 보건 당국 신뢰 확보의 핵심

### 1.3 기술 스택

| 계층 | 기술 |
|------|------|
| Frontend | Streamlit (현재) → Next.js 14 + Deck.gl (목표) |
| Backend | FastAPI + SQLAlchemy (async) |
| Pipeline | Kafka KRaft + httpx + pdfplumber |
| ML | TFT (PyTorch Forecasting) + Autoencoder + RAG-LLM |
| DB | TimescaleDB + Qdrant (벡터) |
| Infra | Docker Compose (개발) / Kubernetes GKE (배포) |
| CI/CD | GitHub Actions (ruff + mypy + pytest) |

## 4. 개선해야 할 부분 (기술적 검증)

### 4.1 CRITICAL — 서비스화 필수

#### (1) KOWAS PDF 자동화
**현재**: 수동으로 PDF 다운로드 후 pdfplumber로 파싱
**문제**: 서비스화 시 매주 사람이 PDF를 넣어줘야 함
**해결**: Selenium/Playwright로 KOWAS 사이트 자동 크롤링 → PDF 다운로드 → 파싱 자동화
**리스크**: KOWAS 사이트 구조 변경 시 크롤러 깨짐 → 모니터링 필수

#### (2) Kafka Consumer 부재
**현재**: Producer만 구현 (데이터를 Kafka에 넣기만 함)
**문제**: Kafka → TimescaleDB 적재 로직이 없음
**해결**: Kafka Consumer 그룹 구현 + DB INSERT 배치 처리

#### (3) TFT 모델 학습 데이터 부족
**현재**: 2024-25 시즌 26주 데이터만
**문제**: TFT는 최소 수백~수천 시점 데이터 필요. 26주로는 과적합 위험
**해결**:
  - 과거 시즌 데이터 역추적 수집 (2019-2024)
  - 데이터 증강 (시계열 augmentation)
  - 또는 TFT → 더 가벼운 모델 (ARIMA + 앙상블) 병행

#### (4) 실시간성 vs 주간 갱신 간극
**현재**: 3개 Layer 모두 주 1회 갱신
**문제**: "실시간 조기경보"라고 하기엔 갱신 주기가 김
**해결**: L3 (검색)은 일간 갱신 가능, L1은 주간이 한계, L2도 주간
**제안**: "주간 경보 + 일간 검색 트렌드 보조 신호" 구조로 명확히 재정의

### 4.2 IMPORTANT — 품질 향상

#### (5) 지역 세분화 부족
**현재**: 서울시 구 단위 지도는 있으나 데이터는 전국 단위
**문제**: 지도 위의 구별 위험도가 시뮬레이션 데이터
**해결**: KOWAS 지역별 데이터 수집 + 네이버 API 지역 파라미터 활용

#### (6) 모델 해석가능성(Explainability) 보강
**현재**: TFT attention weights만 제공
**문제**: 보건 당국이 "왜 경보인지" 설명 요구
**해결**: SHAP values + Layer별 기여도 시각화 + RAG 리포트에 근거 명시

#### (7) 다중 질병 확장성
**현재**: 인플루엔자 전용
**문제**: 캡스톤에서 확장성 어필 약함
**해결**: 코로나/RSV/노로바이러스 등 다른 질병용 Layer 추가 가능 구조 설계

### 4.3 NICE TO HAVE

#### (8) 모바일 접근성
- PWA 또는 React Native 앱
- 푸시 알림으로 경보 전달

#### (9) 공개 API
- 외부 연구자/기관이 활용할 수 있는 REST API 공개
- API 사용량 제한 + 문서화

## 5. 벤치마킹 분석

### 5.1 국내외 유사 시스템

| 시스템 | 운영 주체 | 데이터 소스 | 선행시간 | 한계 |
|--------|----------|------------|---------|------|
| **KCDC 감염병 감시** | 질병관리청 | 의료기관 신고 | 0일 (후행) | 보고 지연, 비의료 신호 미활용 |
| **Google Flu Trends** | Google (2008-2015) | 검색어 | 1~2주 | 과대추정 문제로 2015 폐지 |
| **KOWAS** | 환경부 | 하수 바이러스 | 2~3주 | PDF 보고서만 제공, API 없음 |
| **Biobot Analytics** (미국) | 스타트업 | 하수 | 2~3주 | 미국 한정, 유료 |
| **EIOS** (WHO) | WHO | 뉴스/SNS | 수일 | 텍스트 마이닝, 정량 예측 아님 |
| **BlueDot** (캐나다) | 스타트업 | 항공+뉴스+기후 | 수일 | B2B, 비공개 |

### 5.2 우리 시스템의 차별점

| 차별점 | 설명 |
|--------|------|
| **3-Layer 교차검증** | Google Flu Trends의 단일 신호 실패를 극복. 3개 독립 신호 교차로 오경보 0건 달성 |
| **하수 데이터 활용** | 국내에서 KOWAS 데이터를 ML 파이프라인에 통합한 사례 거의 없음 |
| **오픈소스 + 지자체 배포 가능** | Biobot/BlueDot은 유료 B2B. 우리는 오픈소스로 지자체 직접 도입 가능 |
| **RAG-LLM 경보 리포트** | 단순 수치가 아닌, 역학 근거 기반 자연어 리포트 자동 생성 |
| **한국 특화** | 네이버 API + KOWAS + 기상청 — 국내 데이터 소스에 최적화 |

### 5.3 학술 근거

- Keshaviah et al. (2023) "Wastewater-based surveillance can serve as a leading indicator of COVID-19" — Nature Communications
- Google Flu Trends 실패 분석: Lazer et al. (2014) Science

| 오경보 없이 경보 가능한가? | ✅ 검증됨 | Precision 1.00 |
| TFT가 이 데이터에 적합한가? | ⚠️ 부분 | 코드 완성, 실학습 미완 — 데이터 부족 우려 |
| RAG-LLM이 유용한 리포트를 생성하는가? | ⚠️ 부분 | 프롬프트 완성, 실제 역학 문서 임베딩 미완 |
| Kafka 파이프라인이 안정적인가? | ⚠️ 부분 | Producer 완성, Consumer 미완 |

## 9. 개발 로드맵 (8주)

### 9.1 운영 원칙

- 각 주차는 `구현`, `검증`, `문서화`를 동시에 완료해야 함
- 완료 기준은 코드 작성이 아니라 `테스트/로그/시연 가능 상태`까지 포함
- 발표용 저장소와 실제 실행 상태가 어긋나지 않도록 README, 회의자료, 데모 데이터를 함께 갱신
